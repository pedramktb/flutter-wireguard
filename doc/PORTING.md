# Porting blueprint: iOS, macOS

> Windows landed in 0.2.0 — see [`README.md` § Windows](../README.md#windows)
> for the runtime architecture and [`WINDOWS_SETUP.md`](WINDOWS_SETUP.md) for
> the local-test setup. The blueprint kept below targets the remaining Apple
> platforms.

This document describes how to bring the remaining platforms up to parity with Android, Linux and Windows. The Pigeon contract in [pigeons/messages.dart](pigeons/messages.dart) is the source of truth — every platform must implement the same `WireguardHostApi` and emit `WireguardFlutterApi.onTunnelStatus` events.

## Contract recap

Each platform must implement:

| Method | Semantics |
|---|---|
| `start(name, config)` | Bring tunnel up. Throw `PlatformException(START_FAILED, ...)` on failure. |
| `stop(name)` | Bring tunnel down. No-op when already down. |
| `status(name)` | Return `TunnelStatus { name, state, rx, tx, handshake_ms }`. Throw if unknown. |
| `tunnelNames()` | Names of all known tunnels (including DOWN ones started this session). |
| `backend()` | `BackendInfo { kind: kernel\|userspace\|unknown, detail }`. |
| Push: `onTunnelStatus(status)` | Fired on state changes and ~1 Hz stats ticks while UP. |

Invariants every backend must uphold:

1. **Multi-tunnel**: keep a map keyed by name; do not assume one global tunnel.
2. **Validation**: tunnel name regex `^[A-Za-z0-9_=+.-]{1,15}$`; reject anything else *before* it reaches a shell, kernel API, or filesystem path.
3. **Config secrecy**: any on-disk config must be `0600` (POSIX) / equivalent ACL (Windows), in a per-user runtime dir, deleted on `stop`.
4. **Events on the platform thread**: marshal back to the platform/UI thread before calling the FlutterApi, exactly as the Linux GObject impl does with `g_idle_add` and Android does with `Handler(Looper.getMainLooper())`.
5. **No new shell-outs without `argv` arrays**. Never `system()`/`Runtime.exec(String)`.

## iOS / macOS (shared Darwin source)

The `pubspec.yaml` already declares `sharedDarwinSource: true` and there is a [darwin/Classes/FlutterWireguardPlugin.swift](darwin/Classes/FlutterWireguardPlugin.swift) stub. One Swift file backs both platforms.

### Architecture

Apple platforms cannot run a TUN device inside the app process — the tunnel must live in a **Network Extension (Packet Tunnel Provider)**. The plugin is therefore split into two binaries that talk via `NEVPNManager` + an App Group container:

```
┌──────────────────┐       NEVPNManager        ┌─────────────────────────────┐
│ Flutter app      │ ──── start/stop/status ─▶ │ NEPacketTunnelProvider      │
│  (this plugin)   │                            │  WireGuardKit (linked .dylib)│
└──────────────────┘ ◀── status events ──────  └─────────────────────────────┘
        │                  (Darwin notify /
        │                   App Group file)
        ▼
   WireGuardFlutterApi.onTunnelStatus
```

### Recommended dependency

Use [WireGuardKit](https://git.zx2c4.com/wireguard-apple) as a SwiftPM/CocoaPods dependency. It packages the wireguard-go runtime and a `WireGuardAdapter` that owns the tun handle inside the extension.

### File layout

```
darwin/
  flutter_wireguard.podspec               # add dep on WireGuardKit
  Classes/
    FlutterWireguardPlugin.swift          # implements WireguardHostApi
    TunnelManager.swift                   # wraps NETunnelProviderManager(s)
    BackendDetect.swift                   # always returns .userspace/"WireGuardKit"
    Messages.g.swift                      # generated by pigeon (add swiftOut:)
example/ios/PacketTunnel/                 # Network Extension target (host app)
example/macos/PacketTunnel/               # Network Extension target (host app)
  PacketTunnelProvider.swift              # subclasses NEPacketTunnelProvider
```

The Network Extension targets live in the **example app**, not in the plugin — Apple requires extensions to be embedded by the host application. The plugin must document the Xcode steps in its README (entitlements, App Group, signing).

### Required entitlements (host app + extension)

- `com.apple.developer.networking.networkextension` → `["packet-tunnel-provider"]`
- `com.apple.security.application-groups` → `["group.<your.bundle>.wireguard"]` (used to hand the config blob from app → extension)
- macOS only: `com.apple.security.app-sandbox` if sandboxed, plus the `system-extension` flavour if you ship a System Extension instead of an App Extension.

### Implementation notes per Pigeon method

- `start(name, config)`:
  1. Save the raw config string to the App Group container at `tunnels/<name>.conf`.
  2. `NETunnelProviderManager.loadAllFromPreferences` → reuse or create one whose `localizedDescription == name`.
  3. Set `protocolConfiguration = NETunnelProviderProtocol()` with `providerBundleIdentifier`, `serverAddress = "WireGuard"`, `providerConfiguration = ["tunnelName": name]`.
  4. `saveToPreferences` → `connection.startVPNTunnel()`.
  5. Resolve the Pigeon completion only after `NEVPNStatusConnected` (or reject on `NEVPNStatusDisconnected` within ~10 s).
- `stop(name)`: locate the manager, `connection.stopVPNTunnel()`, then `removeFromPreferences()` if you want the row gone.
- `status(name)`: read `connection.status` for state; for byte counters call `WireGuardAdapter.getRuntimeConfiguration()` *inside the extension* and pipe the result back via `NEVPNConnection.fetchLastDisconnectError`-style XPC, or write the latest stats to a shared file every second from the extension and read here.
- `tunnelNames()`: enumerate `NETunnelProviderManager.loadAllFromPreferences`.
- `backend()`: always `BackendKind.userspace`, `detail = "WireGuardKit (\(WireGuardAdapter.version))"`.
- Events: subscribe to `NEVPNStatusDidChange` notifications and forward to `WireguardFlutterApi.onTunnelStatus(...)` on the main queue. Run a 1 Hz `DispatchSourceTimer` on UP tunnels to read the shared stats file.

### Pigeon

Add to `@ConfigurePigeon`:

```dart
swiftOut: 'darwin/Classes/Messages.g.swift',
swiftOptions: SwiftOptions(),
```

Then `dart run pigeon --input pigeons/messages.dart`.

### Tests

- `darwin/Tests/TunnelManagerTests.swift` — XCTest with a fake `NETunnelProviderManager` factory, exercising the name-validation + status-mapping helpers.
- Stats parser test independent of NetworkExtension (so it runs on plain macOS CI).
- Extend [example/integration_test/plugin_test.dart](example/integration_test/plugin_test.dart) — `backend()` and `tunnelNames()` already work without an actual VPN.

## Windows

There is already a substantial Windows skeleton (`config_writer`, `service_control`, `utils`, bundled `wireguard.dll` + `tunnel.dll` for amd64/arm64). It needs to be put behind the Pigeon API and made multi-tunnel.

### Architecture

Use the official [wireguard-windows](https://git.zx2c4.com/wireguard-windows) embeddable model:

```
┌────────────────────┐      Service Control Manager      ┌──────────────────────┐
│ Flutter app        │ ─── CreateService / StartService ▶│ WireGuard service     │
│ (this plugin C++)  │                                    │  tunnel.dll hosts the │
│   wireguard.dll ◀──│ ──── ring0 IOCTLs (stats) ────────│  TUN driver           │
└────────────────────┘                                    └──────────────────────┘
        ▲                                                           │
        └──────── WTSRegisterSessionNotification / SCM events ──────┘
```

`tunnel.dll` is loaded by the **service** (not the app), so each tunnel is an isolated `Windows Service` named e.g. `WireGuardTunnel$<name>`. The app process only:

- Writes `<name>.conf.dpapi` to `%PROGRAMDATA%\WireGuard\Configurations\` (DPAPI-encrypted, ACL'd to LocalSystem + admins),
- Calls `wireguard.dll` exports (`WireGuardOpenAdapter`, `WireGuardGetConfiguration`) for stats,
- Manages services via `OpenSCManager` / `CreateServiceW`.

### File layout (refactor)

```
windows/
  CMakeLists.txt
  flutter_wireguard_plugin_c_api.{h,cpp}        # keep — Flutter entry point
  flutter_wireguard_plugin.{h,cpp}              # implement Pigeon HostApi here
  messages.g.{h,cpp}                            # generated (cppHeaderOut/cppSourceOut)
  tunnel_manager.{h,cpp}                        # multi-tunnel orchestration
  service_control.{h,cpp}                       # already exists — extend for per-tunnel names
  config_writer.{h,cpp}                         # already exists — switch to DPAPI + per-tunnel path
  wireguard_dll.{h,cpp}                         # thin LoadLibrary wrapper around bundled DLLs
  utils.{h,cpp}                                 # name validation, wide/utf8 conversion
  lib/                                          # bundled wireguard.dll / tunnel.dll (already present)
```

Add elevation/UAC manifest to the example app — `CreateServiceW` requires `SeCreateServicePrivilege` (admin). The plugin must surface `ERROR_ACCESS_DENIED` as `PlatformException(START_FAILED, "Run as administrator")`.

### Pigeon

Pigeon's Windows target *does* compile against the Flutter Windows engine (unlike Linux). Add:

```dart
cppHeaderOut: 'windows/messages.g.h',
cppSourceOut: 'windows/messages.g.cpp',
cppOptions: CppOptions(namespace: 'flutter_wireguard'),
```

### Implementation notes per Pigeon method

- `start(name, config)`:
  1. Validate name (same regex as Linux).
  2. DPAPI-encrypt the config; write `%PROGRAMDATA%\flutter_wireguard\<name>.conf.dpapi` with an ACL granting only `LocalSystem` + `BUILTIN\Administrators`.
  3. `OpenSCManager(SC_MANAGER_ALL_ACCESS)`; `CreateServiceW("WireGuardTunnel$<name>", ..., binaryPath = "<our exe> /service <name>")` where `<our exe>` is a tiny helper that calls `WireGuardTunnelService(<name>.conf)` from `tunnel.dll` — same approach the official wireguard-windows GUI uses. The example app ships this helper alongside the Flutter binary.
  4. `StartServiceW`; complete the Pigeon call when SCM reports `SERVICE_RUNNING` (or fail on stop).
- `stop(name)`: `ControlService(SERVICE_CONTROL_STOP)` → `DeleteService` → delete the encrypted config file.
- `status(name)`: `WireGuardOpenAdapter(name)` → `WireGuardGetConfiguration` to extract last-handshake + rx/tx (sum across peers). Map SCM `SERVICE_RUNNING` → `TunnelState.up`, `STOP_PENDING/START_PENDING` → `toggle`, anything else → `down`.
- `tunnelNames()`: `EnumServicesStatusExW` filtered by name prefix `WireGuardTunnel$`.
- `backend()`: always `BackendKind.userspace`, `detail = "wireguard-nt + tunnel.dll"` (or `"wireguard-go"` if you decide to ship that variant).
- Events: subclass an `IServiceNotify`-based watcher, plus a 1 Hz `SetTimer` polling `WireGuardGetConfiguration` for active tunnels. Marshal back to the Flutter UI thread with `flutter::TaskRunner::PostTask`.

### Tests

- gtest under `windows/test/` mirroring [linux/test/](linux/test/):
  - `name_validation_test.cpp` (shares logic with Linux — consider extracting to a `common/` header).
  - `config_writer_test.cpp` — round-trip plain → DPAPI → plain, verify ACL bits with `GetSecurityInfo`.
  - `service_control_test.cpp` — uses a fake `IScm` interface so the test does not actually touch the real SCM.
- Hook into [.github/workflows/ci.yml](.github/workflows/ci.yml): add a `windows-latest` job that runs `flutter build windows --debug` in `example/` and `cmake --build` + `ctest` for the native tests.

## Cross-cutting work

1. **Shared name validator.** Extract `IsValidName` from [linux/wg_backend.cc](linux/wg_backend.cc) into a header used by both Linux and Windows; mirror the same logic in Swift and Kotlin (Android already has it implicitly via `Tunnel` constructor — make it explicit and unit-test it).
2. **Status JSON / Pigeon mapping tests** are already exemplary on Android ([android/src/test/kotlin/com/pedramktb/flutter_wireguard/StatusJsonTest.kt](android/src/test/kotlin/com/pedramktb/flutter_wireguard/StatusJsonTest.kt)). Add the Swift/Windows equivalents — this is the cheapest test that catches enum-drift between platforms.
3. **Integration test** in [example/integration_test/plugin_test.dart](example/integration_test/plugin_test.dart) already exercises `backend()`, `tunnelNames()`, and an expected-fail `start`. These should pass on every new platform with no changes — run them in CI on a self-hosted macOS / Windows runner before declaring parity.
4. **Docs**: when a platform flips ✅, update the table in [README.md](README.md), add a `### iOS` / `### macOS` / `### Windows` "Platform notes" section covering entitlements / admin / driver requirements, and add the platform to the example's run instructions.
5. **Versioning**: bump to `0.2.0` when adding the first new platform; the Pigeon contract is wire-stable, so `0.1.x` patch releases stay compatible.

## Suggested order

1. **Windows** — most code already exists; wiring it through Pigeon + multi-tunnel + DPAPI is mechanical.
2. **macOS** — Network Extension story is identical to iOS but easier to debug locally.
3. **iOS** — same Swift code as macOS thanks to `sharedDarwinSource`; main extra work is the example project's entitlements + provisioning profiles.
